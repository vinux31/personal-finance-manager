## CRITICAL

### C-01 — `useProcessRecurring` race vs `mark_bill_paid` creates duplicate transactions
**File:** `src/hooks/useProcessRecurring.ts:13-47` (+ `src/db/recurringTransactions.ts:138-151`)
**Category:** Bug / Data Integrity

**Description:** Setiap user buka tab Transaksi (`TransactionsTab` line 45 panggil `useProcessRecurring()`), hook ini memuat semua `recurring_templates`, lalu untuk setiap template yang `next_due_date <= today` ia melakukan **`createTransaction(...)` lalu `updateRecurringTemplate(... next_due_date: nextDue)`** secara non-atomic. Tidak ada idempotency guard mirip `mark_bill_paid` (yang men-cek `bill_payments` row pada `paid_date` sebelum insert).

Skenario reproduksi (telah terjadi di workflow tagihan):
1. User klik "Lunas" pada `UpcomingBillsPanel` → `mark_bill_paid` insert `transactions` + `bill_payments` + advance `next_due_date`. Atomic, idempotent.
2. Tab Transaksi di-mount (atau user pindah ke Transaksi setelah Dashboard): `useProcessRecurring` baca template yang **belum**-nya di-refresh oleh React Query (cache stale, `next_due_date` masih nilai lama yang `<= today`). Hook akan **insert duplikat transaksi** dengan tanggal `next_due_date` lama.

Bukti:
- Tidak ada kueri `bill_payments` di `useProcessRecurring` — hanya `is_active && next_due_date <= today`.
- `nextDueDate(due, t.frequency)` versi TS dipakai di hook, sedangkan `mark_bill_paid` pakai `next_due_date_sql` di SQL. Dua jalur tulis — `useProcessRecurring` adalah jalur lawas yang masih dipertahankan untuk income-recurring (Gaji), namun tetap aktif untuk expense.
- React Query staleTime default = 60s (`src/lib/queryClient.ts:7`). Setelah `mark_bill_paid` mutate, `onSettled` hanya invalidate `['recurring-templates']`, tapi `useProcessRecurring` punya jendela refetch + render race.

**Impact:** Duplikat baris di `transactions` (double-debit di laporan). Tidak ada constraint UNIQUE (template_id, date) di table `transactions`, jadi DB menerima keduanya. User akan melihat tagihan tercatat 2x di Transaksi list dan total expense bulan inflated.

**Fix:**
- Pendekatan tercepat — sebelum `createTransaction` di hook, cek `bill_payments` dengan `(recurring_template_id = t.id AND paid_date = due AND user_id = uid)`. Skip jika sudah ada.
- Pendekatan kanonis — refactor hook untuk panggil `mark_bill_paid` RPC saja (yang sudah punya idempotency guard di `0014_mark_bill_paid.sql:84-91`). Untuk income templates yang tidak ada di view `upcoming_bills_unpaid`, buat RPC paralel `mark_income_received` atau extend `mark_bill_paid` agar accept `type='income'`.

```ts
// src/hooks/useProcessRecurring.ts — patch minimal
for (const t of overdue) {
  let due = t.next_due_date
  let iterations = 0
  while (due <= today && iterations < 12) {
    // Idempotency guard — skip jika sudah dibuat (oleh mark_bill_paid atau run sebelumnya)
    const { data: existing } = await supabase
      .from('bill_payments')
      .select('id')
      .eq('recurring_template_id', t.id)
      .eq('paid_date', due)
      .eq('user_id', uid)
      .maybeSingle()
    if (!existing) {
      await createTransaction({ date: due, type: t.type, ... })
      // Plus insert bill_payments untuk maintain audit trail
    }
    due = nextDueDate(due, t.frequency)
    iterations++
  }
  await updateRecurringTemplate(t.id, { ...t, next_due_date: due })
}
```

### C-02 — `withdrawFromGoal` tidak memperhitungkan investasi yang ter-link → user bisa "tarik" dana yang tidak ada
**File:** `src/db/goals.ts:106-126`, `src/components/AddMoneyDialog.tsx:73-74`, `src/tabs/GoalsTab.tsx:121-125`
**Category:** Bug / Data Integrity

**Description:** UI menampilkan `totalCurrent = goal.current_amount + investedAmount` (cash + slice investasi yang dilink). Saat user buka "Tarik Dana", dialog hanya menampilkan `goal.current_amount` (cash) sebagai "Saldo kas tersedia" — itu benar untuk batas withdraw, tapi:
1. `withdrawFromGoal` mengecek `amount > goal.current_amount` → throw "Dana tidak cukup". Itu masih cocok.
2. **Tetapi** UI label di card Goals menampilkan `formatRupiah(totalCurrent)` di `dari … target …`. User dengan goal `current_amount = 1jt + investasi 50jt = 51jt` akan **terlihat** sudah mencapai 51jt, lalu mencoba tarik 30jt dari "tabungan", dan sistem akan menolak dengan pesan kriptik "Dana tidak cukup atau data sudah berubah" (line 125). Tidak ada penjelasan bahwa "investasi tidak bisa ditarik dari sini".

Lebih kritis lagi: pada `addMoneyToGoal` RPC (migration 0006:225-261), `v_new_status := 'completed' WHEN v_new_amount >= v_target`. Karena goal status di-mark completed berdasarkan `current_amount` SAJA (cash), tapi UI menggunakan `totalCurrent` (cash + investasi) untuk bar progress, **goal yang sebenarnya sudah completed via investasi tidak akan auto-mark completed** sampai user manual tambah cash. Inkonsistensi UI vs source-of-truth DB.

**Impact:** UX confusing — user lihat 100% bar tapi badge masih "Aktif"; pesan error tidak actionable saat tarik gagal; juga risiko user bingung kenapa "uangnya" hilang.

**Fix:**

- Update `add_money_to_goal` RPC agar memperhitungkan `goal_investments` saat menentukan `completed`, ATAU ubah `goalProgress`/`StatusBadge` untuk konsisten pakai cash-only `current_amount`. Dokumentasikan keputusan di README/Panduan.
- Pesan error withdraw: tambahkan info "Hanya saldo kas yang bisa ditarik. Saldo investasi: Rp …".

### C-03 — Edge function `fetch-prices` tetap unauthenticated + CORS `*`
**File:** `supabase/functions/fetch-prices/index.ts:3-78`
**Category:** Security

**Description:** Verified masih open. Tidak ada `Authorization` check; `'Access-Control-Allow-Origin': '*'`. Setiap user yang tahu URL Supabase project + nama function bisa drain quota Yahoo Finance dan metals.dev (paid). `METALS_DEV_API_KEY` aman di sisi server, tapi quota usage adalah biaya nyata.

Karena ini sudah _tracked_ di CONCERNS.md tetapi **belum dimitigasi** dan saya verifikasi masih ada di code current (line 3, 30-78), saya promote ke **Critical** karena: (a) sangat trivial untuk exploit (anon key + nama function publik di bundle JS), (b) impact = financial cost, (c) fix-nya straightforward.

**Impact:** Quota abuse, cost overrun pada metals.dev, potential rate-limit dari Yahoo (akan break refresh harga untuk user legit).

**Fix:**
```ts
// supabase/functions/fetch-prices/index.ts (top of serve handler, after OPTIONS)
const authHeader = req.headers.get('Authorization')
if (!authHeader?.startsWith('Bearer ')) {
  return new Response(JSON.stringify({ error: 'Unauthorized' }), {
    status: 401,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
}
// Optional: validate JWT against Supabase
const token = authHeader.replace('Bearer ', '')
const supabaseAdmin = createClient(Deno.env.get('SUPABASE_URL')!, Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!)
const { data: { user }, error: authErr } = await supabaseAdmin.auth.getUser(token)
if (authErr || !user) return new Response('Unauthorized', { status: 401, headers: corsHeaders })

// Plus tighten CORS:
const corsHeaders = {
  'Access-Control-Allow-Origin': Deno.env.get('ALLOWED_ORIGIN') ?? 'https://kantongpintar.app',
  'Access-Control-Allow-Headers': 'authorization, x-client-info, apikey, content-type',
}
```

### H-03 — `goal_investments.allocation_pct` cross-row enforcement masih client-side only
**File:** `supabase/migrations/0005_goal_investments.sql:5`, `src/components/LinkInvestmentDialog.tsx:46-84`, `src/db/goalInvestments.ts:21-33`
**Category:** Data Integrity / Security

**Description:** CHECK constraint `allocation_pct > 0 AND allocation_pct <= 100` hanya per-row. Logika "total allocation across goals untuk satu investasi tidak boleh > 100%" hanya di client (`LinkInvestmentDialog.tsx:46-51,77-84`). Dua tab/race → over-allocation lewat. Tidak ada DB trigger.

**Impact:** Investasi yang dialokasikan > 100% akan menyebabkan `Goals` summary di Dashboard double-count + bug nominal di `formatRupiah(currentValue(inv) * a.allocation_pct / 100)` (Goals card line 175). `Net Worth` di KekayaanTab juga miss — investasi penuh dihitung sekaligus _juga_ dialokasikan ke goal.

**Fix:** Tambah DB trigger atau RPC `link_investment_to_goal`:
```sql
CREATE OR REPLACE FUNCTION enforce_goal_investment_total()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  v_total NUMERIC;
BEGIN
  SELECT COALESCE(SUM(allocation_pct), 0)
  INTO v_total
  FROM goal_investments
  WHERE investment_id = NEW.investment_id
    AND id != COALESCE(NEW.id, -1);
  IF v_total + NEW.allocation_pct > 100 THEN
    RAISE EXCEPTION 'Total alokasi investasi % melebihi 100%% (sudah %, mau tambah %)',
      NEW.investment_id, v_total, NEW.allocation_pct;
  END IF;
  RETURN NEW;
END;
$$;
CREATE TRIGGER goal_investments_total_check
  BEFORE INSERT OR UPDATE ON goal_investments
  FOR EACH ROW EXECUTE FUNCTION enforce_goal_investment_total();
```

### H-04 — `profiles_select` & `allowed_emails_select` policies = `USING (true)` (info disclosure)
**File:** `supabase/migrations/0006_multi_user.sql:15, 29`
**Category:** Security

**Description:** Tracked di CONCERNS, **belum di-fix di code saat ini**. Setiap authenticated user bisa baca semua `profiles` (termasuk `is_admin` boolean) dan semua `allowed_emails`. Verified — masih ada di migration 0006 dan tidak ada migration berikutnya (0007-0016) yang merubah policy ini.

**Impact:** Bila app dibuka publik (atau allowlist diperluas), user bisa enumerate admin + steal email list lewat REST endpoint langsung (bypass UI). Privacy leak.

**Fix:**
```sql
-- Migration baru 0017_tighten_rls.sql
DROP POLICY profiles_select ON profiles;
CREATE POLICY profiles_select ON profiles FOR SELECT TO authenticated
  USING (id = auth.uid() OR is_admin());

DROP POLICY allowed_emails_select ON allowed_emails;
CREATE POLICY allowed_emails_select ON allowed_emails FOR SELECT TO authenticated
  USING (is_admin());
```

### H-06 — `aggregate_by_period` & `aggregate_by_category` SECURITY DEFINER tanpa cek admin saat `p_user_id` non-NULL
**File:** `supabase/migrations/0006_multi_user.sql:169-222`
**Category:** Security

**Description:**
```sql
WHERE user_id = COALESCE(p_user_id, auth.uid())
```
RPC `SECURITY DEFINER` — bypass RLS. Setiap caller bisa pass `p_user_id = '<uid_orang_lain>'` dan dapat agregat keuangan user lain. Tidak ada cek `is_admin()` atau `p_user_id = auth.uid()`.

**Impact:** Authenticated user bisa: panggil `supabase.rpc('aggregate_by_period', { p_user_id: <admin_uid> })` → langsung dapat ringkasan income/expense admin per bulan. Setara IDOR pada laporan keuangan.

**Fix:**
```sql
CREATE OR REPLACE FUNCTION aggregate_by_period(...)
... AS $$
DECLARE
  v_target_uid UUID := COALESCE(p_user_id, auth.uid());
BEGIN
  IF v_target_uid != auth.uid() AND NOT is_admin() THEN
    RAISE EXCEPTION 'Akses ditolak';
  END IF;
  -- ... query menggunakan v_target_uid ...
END;
$$;
```
Apply juga ke `aggregate_by_category`.
