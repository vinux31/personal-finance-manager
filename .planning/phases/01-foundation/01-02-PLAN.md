From 0011_pension_simulations.sql (canonical RLS pattern to follow):
```sql
ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users manage own ..."
  ON <table> FOR ALL
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);
```

D-06 mandates `OR is_admin()` on USING (read) but NOT on WITH CHECK (write) — so the final pattern to use is:
```sql
USING  (auth.uid() = user_id OR is_admin())
WITH CHECK (auth.uid() = user_id)
```

    Important FK semantics (finding #6 in planning context):

    - `amount`: `CHECK (amount > 0)` — matches convention from 0010's transaction amount check.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase PostgREST API → DB | Every SELECT/INSERT/UPDATE/DELETE flows through RLS; `auth.uid()` is derived from the signed JWT |
| Admin user → non-admin data | Admin has read-any via `is_admin()` but MUST NOT be able to forge writes on behalf of another user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | Cross-user SELECT leaks net worth data | mitigate | RLS `USING (auth.uid() = user_id OR is_admin())` on all 4 tables — verified in Task 3 step 4 |
| T-02-02 | Elevation of Privilege | Admin forges writes as another user | mitigate | `WITH CHECK (auth.uid() = user_id)` has NO `is_admin()` clause — verified in Task 3 step 6 |
| T-02-03 | Tampering | Negative or zero payment amount | mitigate | `CHECK (amount > 0)` on bill_payments.amount |
| T-02-04 | Tampering | Invalid account/liability type | mitigate | `CHECK (type IN (...))` allowlist on both net_worth_accounts.type and net_worth_liabilities.type |
| T-02-05 | Denial of Service | Computed column causes insert failure | accept | GENERATED ALWAYS AS STORED is well-supported in Supabase/PostgreSQL 15+; verified in Task 3 step 7 |
| T-02-06 | Repudiation | No audit log for bill_payment deletions | accept | Out of scope for v1.0; `created_at` provides basic temporal record |
| T-02-07 | Tampering | Orphaned bill_payment when recurring_template deleted | mitigate | `ON DELETE CASCADE` on recurring_template_id — payment history discarded with template (per design, finding #6) |
</threat_model>
