<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase (PostgREST) | User-supplied `uid` (from `useTargetUserId()`) crosses into the DB query; all filter predicates are attacker-influenceable via URL/session state |
| Supabase RLS policy → recurring_templates table | Rows must be scoped to `auth.uid() = user_id`; any bypass leaks another user's bills |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | `listUpcomingBills` Supabase query | mitigate | Table `recurring_templates` already has `auth.uid() = user_id` RLS policy (migration 0010). The `.eq('user_id', uid)` predicate is a defense-in-depth filter but RLS is the primary guard — verified by existing usage of `listRecurringTemplates` in the codebase. |
| T-03-02 | Tampering | `endOfMonth` parameter (string from client) | mitigate | `endOfMonth` is computed client-side in `useUpcomingBills` from `new Date()` — never user-supplied. PostgREST `.lte()` parameter-binds the value (no SQL injection vector). If a malicious client crafts a different endOfMonth, RLS still limits rows to `auth.uid()` ownership; worst case is seeing more of *their own* bills, which is not a security issue. |
| T-03-03 | Elevation of Privilege | Admin view-as via `useTargetUserId` | mitigate | `useTargetUserId` is the codebase-wide admin-view hook already used by `useRecurringTemplates`. Admin permission check happens inside that hook (verified in Phase 1/2). No new privilege path is introduced by this plan. |
| T-03-04 | Denial of Service | Unbounded query result set | accept | Rationale: a single user's active recurring templates for one month is O(tens). No pagination added — row count is naturally bounded. Re-evaluate in v2 if users exceed ~100 templates. |
| T-03-05 | Repudiation | Read-only data fetch | accept | Rationale: read-only query, no state mutation, no audit requirement for list operations (logged by Supabase at platform layer). |
| T-03-06 | Spoofing | Session identity forgery | accept | Out of scope — handled globally by Supabase Auth / JWT validation; not introduced or weakened by this plan. |
</threat_model>
