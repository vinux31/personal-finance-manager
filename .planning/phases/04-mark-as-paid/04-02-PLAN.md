Create the file `supabase/migrations/0015_upcoming_bills_unpaid_view.sql` with EXACTLY this content:

```sql
-- ============================================================
-- 0015_upcoming_bills_unpaid_view: Sisa Aman D-03 — exclude bills
-- already paid in the current month. First VIEW in the project.
-- Feeds listUpcomingBills() via a simple .from('upcoming_bills_unpaid')
-- swap and automatically keeps client arithmetic correct.
-- ============================================================

-- security_invoker = true (PG 15+) makes the view inherit the CALLER's RLS
-- policies on the underlying tables (recurring_templates + bill_payments).
-- Without this flag, Postgres defaults to security_definer semantics where
-- the view runs as its owner (typically a superuser), bypassing RLS entirely.
-- RESEARCH Pitfall 7 — mandatory.
CREATE OR REPLACE VIEW upcoming_bills_unpaid
WITH (security_invoker = true)
AS
SELECT
  t.id,
  t.user_id,
  t.name,
  t.type,
  t.category_id,
  t.amount,
  t.note,
  t.frequency,
  t.next_due_date,
  t.is_active,
  t.created_at
FROM recurring_templates t
WHERE t.is_active = true
  AND t.type = 'expense'
  AND NOT EXISTS (
    SELECT 1
    FROM bill_payments bp
    WHERE bp.recurring_template_id = t.id
      AND bp.user_id = t.user_id
      AND bp.paid_date >= date_trunc('month', CURRENT_DATE)::DATE
      AND bp.paid_date <  (date_trunc('month', CURRENT_DATE) + INTERVAL '1 month')::DATE
  );

-- Grant SELECT on the view itself (RLS on underlying tables still enforced
-- because of security_invoker = true).
GRANT SELECT ON upcoming_bills_unpaid TO authenticated;
```

Critical values baked in (do NOT change any of these):
- View name: `upcoming_bills_unpaid` (snake_case, matches migration filename stem)
- `WITH (security_invoker = true)` — MANDATORY per Pitfall 7; without this the view leaks rows across users
- FK column is `bill_payments.recurring_template_id` — NOT `template_id` (per 0013 schema)
- Month window: `paid_date >= date_trunc('month', CURRENT_DATE)::DATE AND paid_date < (date_trunc('month', CURRENT_DATE) + INTERVAL '1 month')::DATE` — half-open interval [start-of-month, start-of-next-month), avoids off-by-one on last day of month
- Column projection matches `RecurringTemplate` TS shape + `user_id` (for client `.eq('user_id', uid)` filter) + `created_at` (defensive — matches table shape)
- `GRANT SELECT ... TO authenticated` — mirrors `GRANT EXECUTE` discipline from 0003

DO NOT apply the migration in this task — that happens in Plan 04 (Wave 2 schema push). This task only authors the SQL file.
