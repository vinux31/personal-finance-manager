⚠ **DEVIATION DETECTED** — 4 aggregate functions live instead of 2. Plan 05-01 used `CREATE OR REPLACE FUNCTION` to swap the aggregates from `LANGUAGE sql` to `LANGUAGE plpgsql` while ALSO adding a new arg `p_user_id UUID DEFAULT NULL`. Because PostgreSQL keys function identity on argument signature, the new 4-arg plpgsql versions were created alongside, not replacing, the legacy 3-arg `sql` versions.

### Vulnerability scope of the legacy 3-arg sql versions

```sql
CREATE FUNCTION aggregate_by_period(p_granularity TEXT, p_date_from DATE, p_date_to DATE)
RETURNS TABLE(...)
LANGUAGE sql STABLE SECURITY DEFINER
AS $$
  SELECT ... FROM transactions WHERE (date filters only) GROUP BY ...
$$;
```

- `SECURITY DEFINER` → bypasses RLS on `transactions`.
- Body has **no user_id filter at all**.
- Net effect: any authenticated user calling the 3-arg form via PostgREST gets aggregates of ALL users' transactions (info-disclosure).
- Pre-existing bug from migration 0006:169; Phase 5 claimed to close it but the legacy form remained reachable.

Frontend (`src/db/reports.ts:22-46`) passes all 4 named args including `p_user_id`, so PostgREST resolves UI calls to the new plpgsql function — UI is not impacted. But an attacker with a valid JWT can directly POST a 3-arg form and bypass.

### Patch — `supabase/migrations/0018_drop_legacy_aggregates.sql`

Authored locally and applied via Studio fallback (same path as 0017). Content:

```sql
DROP FUNCTION IF EXISTS public.aggregate_by_period(TEXT, DATE, DATE);
DROP FUNCTION IF EXISTS public.aggregate_by_category(TEXT, DATE, DATE);
```

Pre-DROP safety scan:
- pg_depend → 0 dependencies on the legacy functions
- repo grep → 0 callers using 3-arg form (frontend + tests all use 4 args)
- Behavior post-DROP for any future legacy 3-arg caller: resolves to new plpgsql via `p_user_id DEFAULT NULL` → `COALESCE(p_user_id, auth.uid())` → safe own-data scope

User confirmed `dropped` after paste + Run in Studio.
