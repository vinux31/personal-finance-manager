<phase5_lesson>
**Critical reminder (Phase 5 0017→0018 lesson):**
Phase 6 functions are NEW — no signature collision risk on first deploy. BUT if any future patch ever changes a Phase 6 function signature (e.g. adds `p_dry_run BOOLEAN` to `process_due_recurring`), MUST emit:
```sql
DROP FUNCTION IF EXISTS process_due_recurring(DATE, UUID, INT);
CREATE OR REPLACE FUNCTION process_due_recurring(...new sig...) ...
```
Otherwise PostgreSQL keys function identity on `(name, argument_types)` and creates a second overload — old version stays callable, race + IDOR vector via direct PostgREST 3-arg call. Phase 5's `0018_drop_legacy_aggregates.sql` is the canonical reference for this lesson. Add to 06-VERIFICATION.md as code-review checklist.
</phase5_lesson>
